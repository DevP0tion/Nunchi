# 눈치 (nunchi)

> 작업 강도 보정("적당히") passive skill for Claude Code

"적당히"는 고정 규칙이 아니라 **환경별로 학습되는 판단 기준**이다. 어떤 프로젝트는 테스트 생략을 벌주고, 어떤 프로젝트는 과잉 검증이 시간 낭비다. nunchi는 **예측과 실제가 어긋난 순간**만 프로젝트별 보정 DB(`memory.db`의 `memory` 테이블)에 축적하고, 세션 시작 시 코어를 자동 주입하고 매 메시지마다 관련 항목을 검색 회수해서 검증 깊이 · 테스트 범위 · 리서치 강도 · 리팩토링 범위를 결정하는 기준으로 재귀 개선한다.

CLAUDE.md나 메모리 문서에 경험을 적어 두는 자기개선은 플러그인 없이도 가능하다. nunchi가 더하는 것은 그 과정의 **시스템화**다 — 기록은 hook이 강제하고, 회수는 검색이 보장하고, 모든 규칙은 신뢰도와 반전을 가진 반증 가능한 가설로 관리된다. 차이의 전체 비교는 [문서 기반 자기개선과의 차이](#문서-기반-자기개선과의-차이-claudemd--메모리-문서) 참조.

## 요구사항

- **Bun** 이 PATH에 있어야 한다 (hooks가 `bun <script>.ts` 로 실행됨). 확인: `bun --version`
- hooks·memory server·MCP의 의존성(socket.io, @modelcontextprotocol/sdk 등)은 Bun auto-install이 실행 시점에 자동 해석하므로 별도 `bun install`이 필요 없다.

## 설치

**정식 설치 (필수)** — hooks가 `${CLAUDE_PLUGIN_ROOT}`를 사용하므로 반드시 plugin 시스템으로 설치해야 한다. `.claude/skills/` 폴더에 복사하는 방식은 SKILL.md만 로드되고 **hooks(SessionStart 주입, Stop 점검, server auto-start)가 전혀 동작하지 않는다.**

Claude Code 안에서:

```
/plugin marketplace add DevP0tion/DevP0tion
/plugin install nunchi@devp0tion
```

hooks는 자동 등록되며 **다음 세션 시작부터** 동작한다 (설치한 현재 세션에는 미적용).

**개발/테스트용**: 설치 없이 실행하려면

```sh
claude --plugin-dir /path/to/nunchi
```

(hooks 파일 변경 시에는 dev 모드에서도 `/reload-plugins` 필요)

확인: `claude plugin list` 에 `nunchi` 표시. 새 세션 시작 시 컨텍스트에 `[nunchi]` 주입 메시지가 보이면 정상.

## Codex CLI (ChatGPT) 지원

Codex CLI 지원은 **`codex-support` 브랜치**에서 제공한다 (`.codex-plugin/plugin.json` 매니페스트 포함 — main에는 Claude Code 설정만 둔다). 마켓플레이스에는 해당 브랜치로 고정된 **`nunchi-codex`** 항목이 별도로 등록되어 있다:

```
/plugin marketplace add DevP0tion/DevP0tion
/plugin install nunchi-codex@devp0tion
```

Claude Code와의 차이 등 상세는 해당 브랜치의 README를 참조.

## 왜 만들었나

Claude Code는 세션이 끝나면 "이 프로젝트에서 뭐가 통하고 뭐가 사고 나는지"를 잊는다. nunchi는 그 경험 중 **다음 판단을 바꿀 것만** 남긴다:

- **과잉이었음** — 했는데 불필요했던 검증/방어/추상화 → "용서하는 것"
- **과소였음** — 생략했다가 실제로 문제가 된 것 → "벌주는 것"
- **환경 특이사항** — 문서/상식과 다른 이 환경 고유의 동작 → "환경 특이사항"

예측대로 흘러간 평범한 작업, 일회성 실수는 기록하지 않는다. 다음 작업의 판단을 실제로 바꿀 정보만 남기고, 그렇지 않은 기록은 아예 만들지 않는다 — 기록이 늘수록 회수 품질이 떨어지는 것을 막기 위해서다.

## 문서 기반 자기개선과의 차이 (CLAUDE.md · 메모리 문서)

Claude Code든 Codex든, CLAUDE.md·AGENTS.md·메모리 문서에 경험을 적고 다음 세션에 읽는 자기개선은 플러그인 없이도 이미 가능하다. nunchi는 그 대체재가 아니라, 그중 **작업 강도 판단**이라는 한 영역을 골라 기록·회수·반증을 시스템으로 만든 것이다.

| | 문서 기반 자기개선 (CLAUDE.md, 메모리 문서 등) | nunchi |
|---|---|---|
| 기록 대상 | 제한 없음 — 사실, 선호, 코드 패턴, 프로젝트 지식 전부 | 의도적으로 좁음 — **예측과 실제가 어긋난 순간**만, 작업 강도 판단 전용 |
| 기록 트리거 | 모델 재량 또는 사용자 요청. 모델이 잊으면 유실 | Stop hook이 N턴마다 점검을 **강제** — 기록 기회가 시스템에 의해 보장 |
| 회수 방식 | 전문 통째 주입(항목 수에 비례해 컨텍스트 비용 증가) 또는 모델이 알아서 읽어야 함 | 코어만 상시 주입 + 매 메시지 FTS5 검색 상위 3건 — 항목 수와 무관하게 비용 일정 |
| 회수 보장 | 모델 재량 — 관련 문서를 안 읽고 지나칠 수 있음 | hook이 주입하므로 모델 재량과 무관하게 도달 |
| 규칙의 수명 | 한 번 적히면 영속 — 틀린 규칙도 자동 교정 없음 | 신뢰도 승격 · 반전 규칙 · 정제 — 모든 항목이 반증 가능한 가설 |
| 스키마 | 자유 텍스트 — 근거·날짜·확신 수준이 뒤섞이거나 누락 | 고정 스키마 (규칙 · 근거 · 신뢰도, 3섹션) |
| 실패 모드 | 문서 무한 성장 → 컨텍스트 잠식, 낡은 규칙 잔존, 기록 누락 | 근거 없는 생략 → 반전 규칙으로 방지 |

### 저장 매체의 트레이드오프 — 문서 vs DB

저장 매체 선택에는 양쪽 다 실익이 있다. 문서의 강점은 **사람 쪽 인터페이스**(열람, 편집, git, 팀 공유, 무의존성)이고, DB의 강점은 **기계 쪽 연산**(검색 회수, 동시성, 필드 단위 갱신)이다. nunchi가 DB를 택한 것은 회수 파이프라인(매 메시지 검색 주입)이 설계의 중심이기 때문이고, 문서의 강점 일부는 `doc()` 내보내기와 항목에 내장된 근거·날짜 필드로 완화한다.

| 축 | 문서 (markdown 파일) | sqlite DB (nunchi) |
|---|---|---|
| 사람의 열람·직접 편집 | 유리 — 에디터로 바로 읽고 고침, 잘못된 항목을 사용자가 즉시 삭제 | 불리 — `nunchi_list`/`/nunchi`나 sqlite 클라이언트 필요 (완화: `doc()`이 markdown으로 렌더링) |
| git 버전 관리 | 유리 — diff·blame·리뷰·롤백이 공짜, 규칙의 유래를 git 이력으로 추적 | 불리 — 바이너리라 diff 불가 (완화: 항목 자체에 근거·날짜 필드가 있어 이력을 내장) |
| 팀 공유 | 유리 — 저장소에 커밋하면 끝, 별도 인프라 없음 | 부분적 — 파일 복사는 가능하나 병합 불가, 공유는 external-address 서버 필요 |
| 인프라 의존성 | 유리 — 파일만 있으면 어느 에이전트든 읽음 | 불리 — Bun + memory server 필요, 서버가 죽으면 회수 중단 |
| 검색 | 불리 — 전문 주입이거나 grep 수준, 항목이 늘수록 관련 항목 찾기가 모델 몫 | 유리 — FTS5 BM25 랭킹, 다중 쿼리 OR-병합, 수백 건에서도 상위 N건 정밀 회수 |
| 동시 접근 | 불리 — 여러 프로세스가 같은 파일을 쓰면 유실·충돌 | 유리 — server 단일 소유 + 포트 락으로 구조적으로 안전 |
| 구조적 갱신 | 불리 — "신뢰도 +1", "섹션 이동" 같은 필드 단위 갱신이 텍스트 편집이라 깨지기 쉬움 | 유리 — 스키마가 있어 confirm/reverse/정제가 안전한 연산 |
| 확장 경로 | 불리 — 임베딩·랭킹을 붙이려면 결국 DB화 | 유리 — embedding 컬럼 추가 등 업그레이드 경로 예약됨 |

축적되는 문서 예시:

```markdown
# 보정 — my-project

## 벌주는 것 (반드시 한다)

### [배포: CI 캐시]
- 규칙: lockfile 변경 시 CI 캐시 키를 반드시 확인한다
- 근거: 2026-06-12 캐시 미스매치로 배포 2회 실패
- 신뢰도: 높음(3)

## 용서하는 것 (생략 가능)

### [테스트: 내부 스크립트]
- 규칙: scripts/ 하위 일회성 스크립트는 테스트 생략 가능
- 근거: 2026-06-20 테스트 작성이 스크립트 본체보다 오래 걸렸음
- 신뢰도: 중간(2)
```

## 동작

1. **SessionStart** (startup/resume/clear/compact): 규약 요약 + 코어('벌주는 것' 신뢰도 3+)를 additionalContext로 조용히 주입. `auto-start: true`면 memory server 자동 기동. 기존 calibration.md는 서버 첫 기동 시 DB로 자동 임포트(`.imported`로 보존).
2. **UserPromptSubmit** (매 메시지): 프롬프트 어절로 보정 DB를 검색해 관련 항목 상위 3건만 주입. 결과 0건이면 비용 0. 서버 미기동이면 조용히 통과.
3. **SubagentStart**: 서브에이전트에 규약 + 코어 주입 (SessionStart 주입을 못 받으므로).
4. **모델 재량 (MCP 도구)**: `nunchi_search`(유의어 확장 쿼리 검색) · `nunchi_list`(전량 선별) · `nunchi_record`(예측 어긋남 기록) · `nunchi_update`(재확인·반전·정제).
5. **Stop hook** (백업): 10턴마다 1회 "이번 구간에 예측 어긋남이 있었나?" 점검을 강제. 구간 내에 DB 기록이 이미 있으면 자동 생략.
6. **`/nunchi`**: 수동 호출 시 항목 정제(pruning) 모드.

## 설정 (config)

우선순위: **프로젝트 `.claude/nunchi.json` > plugin userConfig(환경 변수) > 내장 기본값** (키 단위 병합).

전역 설정은 `.claude-plugin/plugin.json`의 `userConfig`로 선언되어 있고, Claude Code plugin 설정 UI에서 값을 지정하면 hook 실행 시 `CLAUDE_PLUGIN_OPTION_<KEY_대문자>` 환경 변수로 주입된다. config 변경은 plugin 재로드 없이 다음 hook 실행부터 반영된다. 손상된 JSON은 무시되고 다음 계층으로 폴백.

| userConfig 키 | 환경 변수 | nunchi.json 키 | 기본값 | 설명 |
|---|---|---|---|---|
| `auto_start` | `CLAUDE_PLUGIN_OPTION_AUTO_START` | `auto-start` | `true` | `true`면 SessionStart 시 memory server 자동 시작 |
| `path` | `CLAUDE_PLUGIN_OPTION_PATH` | `path` | `.claude/nunchi` | 보정 DB(memory.db)가 저장될 폴더 (프로젝트 루트 기준 상대 또는 절대). 초기화 시 없으면 생성 |
| `port` | `CLAUDE_PLUGIN_OPTION_PORT` | `port` | `null` | memory server(Socket.IO) 포트. 미설정 시 memory-config.json의 port(기본 41720) 사용 |
| `external_address` | `CLAUDE_PLUGIN_OPTION_EXTERNAL_ADDRESS` | `external-address` | `null` | 설정 시 로컬 서버 대신 이 주소의 외부 memory server에 연결 (예: `http://192.168.0.10:41720`, 스킴 생략 시 `http://`). 로컬 자동 스폰과 프로젝트 검증 핸드셰이크는 생략 |
| `policy_priority` | `CLAUDE_PLUGIN_OPTION_POLICY_PRIORITY` | `policy-priority` | `null` | ponytail 활성 시 보정 DB와 충돌하면 우선할 쪽 (`nunchi` | `ponytail`)\| `ponytail`). 미결정이면 첫 충돌 시점에 사용자에게 질문 후 프로젝트 nunchi.json에 저장 — 아래 "우선순위 핸드셰이크" 참조 |

프로젝트별 예시 — `{프로젝트}/.claude/nunchi.json`:

```json
{
  "auto-start": false,
  "path": "docs"
}
```

## Memory server

sqlite(`memory.db`)는 server.ts 단일 프로세스만 소유하고, MCP 서버들은 `client.ts`의 `connectMemory()`로 Socket.IO 접속해서 사용한다 — MCP가 여럿 떠도 sqlite 동시 접근 문제가 없다.

- **단일 실행**: 포트 바인딩이 락. 중복 실행하면 `EADDRINUSE` 감지 후 즉시 종료(exit 0).
- **자동 연결**: `connectMemory()`는 `auto-start`와 무관하게 포트에 실행 중인 서버가 있으면 그대로 연결한다. 서버가 없으면 `auto-start: true`일 때만 스폰 후 재접속 (동시 스폰 경쟁은 포트 락이 정리), `false`면 에러.
- **프로젝트 검증 핸드셰이크**: 여러 프로젝트가 같은 포트(기본 41720)를 쓰면 나중에 뜬 쪽이 남의 서버(= 남의 memory.db)에 붙을 수 있다. 이를 막기 위해 `connectMemory()`는 접속 직후 서버의 소유 프로젝트를 확인(`mem:info`)하고, 불일치면 `ProjectMismatchError`를 던진다. 이 에러를 받으면 사용자에게 물어본 뒤 둘 중 하나로 처리한다: ① `connectMemory(projectDir, { force: true })`로 강제 연결(타 프로젝트 db 공유), ② `assignFreePort(projectDir)`로 OS의 빈 포트를 받아 `.claude/nunchi.json`의 `port`에 기록 후 재연결. nunchi.json은 plugin userConfig(환경 변수)보다 우선하므로 즉시 반영된다.
- **설정**: `{path}/memory-config.json` (서버 전용 — 플러그인 config와 별개). `db`(파일명, 기본 `memory.db`), `port`(기본 41720), `host`(바인딩 주소, 기본 `127.0.0.1`), `model`(키워드 보강 모델, 기본 `null`), `modelProvider`(보강 CLI 공급자 `"claude"`/`"codex"`/`"gemini"`, 기본 `"claude"`). 플러그인 config의 `port`가 설정돼 있으면 그쪽이 우선.
- **외부 서버**: 플러그인 config에 `external-address`를 설정하면 로컬 스폰 없이 해당 주소로 연결한다 (명시적 공유 서버이므로 프로젝트 검증 핸드셰이크 생략). 외부에 서비스하는 쪽은 memory-config.json의 `host`를 `0.0.0.0` 등으로 바꿔 바인딩을 연다 — 인증이 없으므로 신뢰할 수 있는 네트워크에서만 열 것.
- **API**: `doc()` / `shutdown()` + `add` / `update` / `remove` / `search(queries[], {section, limit, excludeCore})` / `list` / `core` / `stamp`
- **보정 검색**: `mem:search`는 다중 쿼리 OR-병합 (FTS5 BM25, 3글자 미만·무결과는 LIKE 폴백). 시맨틱 매칭은 쿼리를 확장하는 모델 쪽이 담당한다 — 임베딩 불필요. 필요해지면 `memory` 테이블에 embedding 컬럼을 추가하는 업그레이드 경로가 예약되어 있다.
- **문서 요청**: `doc()`(`mem:doc`)은 보정 DB에서 3섹션 markdown을 렌더링해 반환한다 (없으면 `null`) — external-address 구버전 클라이언트·내보내기 겸용.
- **검색**: FTS5(trigram, BM25 랭킹). 3글자 미만 질의는 LIKE 폴백. 구버전 db는 기동 시 자동 마이그레이션·백필.
- **키워드 보강**: memory-config.json에 `model`을 설정하면(예: `"haiku"`) 보정 기록마다 보강 CLI를 백그라운드로 돌려 유의어 키워드를 생성, 검색 대상에 포함한다. CLI는 `modelProvider`로 선택 — `"claude"`(기본, `claude -p --model <값>`), `"codex"`(`codex exec --model <값>`, 최종 메시지는 `--output-last-message` 임시 파일로 회수), `"gemini"`(`gemini -m <값>`, stdin 파이프 headless). 구현은 `memory/provider/`에 공급자별 파일로 분리 — 새 공급자는 파일 추가 후 `provider/index.ts`의 `PROVIDERS`에 등록. 값이 갱신되면 낡은 키워드는 자동 폐기. `model` 미설정 시 완전 비활성. 기동 시 1회 로드되므로 변경은 memory server 재시작 후 반영. (0.10.0에서 플러그인 config → memory-config.json으로 이동)

## 응용 — 절차형 워크플로 스킬 모음과 상호보완 (예: gstack)

`/ship`, `/qa`, `/review` 같은 절차형 워크플로 스킬 모음은 "무엇을 어떤 순서로 할지"의 **상한**(체크리스트, 게이트, 리뷰 파이프라인)을 제공한다. nunchi는 그 반대 방향 — "이 환경에서 그중 무엇이 실제로 가치 있는지"의 **판단 기준**을 학습한다. 철학은 상반되지만(완전성 최대주의 vs 강도 보정) 층위가 달라 함께 쓰면 서로를 보완한다.

| | 워크플로 스킬 모음 (gstack 등) | nunchi |
|---|---|---|
| 담당 | 절차 — 단계, 게이트, 체크리스트 | 강도 — 단계별 깊이, 생략 가능 여부 |
| 지식 저장 | learnings 등 검색형 저장소 (패턴·함정·선호) | 보정 DB(코어 상시 주입 + 관련 항목 자동 회수) |
| 실패 모드 | 모든 환경에 최대 강도 → 과잉 절차 | 근거 없는 생략 → 반전 규칙으로 방지 |

**응용 방법:**

1. **절차의 강도 판단 기준으로.** 보정 DB는 세션 시작 시 코어가, 매 메시지마다 관련 항목이 자동 회수되므로, 워크플로 스킬이 어떤 단계를 요구할 때 "이 환경에서 이 단계가 가치 있었는가"를 판단하는 근거가 된다. 예: main 직접 커밋 저장소에서 `/ship`의 PR 생성·VERSION 파일 단계가 반복적으로 관행과 불일치했다면 "용서하는 것"에 기록 → 다음 `/ship`부터 근거를 갖고 생략한다.
2. **생략의 안전장치로.** 생략했다가 사고가 나면 반전 규칙에 따라 해당 항목이 즉시 "벌주는 것"으로 이동한다. 워크플로 단계를 설정으로 영구히 끄는 것과 달리, nunchi의 생략은 항상 반증 가능한 가설로 유지된다.
3. **중복 기록은 피한다.** 코드 패턴·함정·환경 지식은 스킬 모음의 learnings에, **작업 강도 판단만** 보정 DB에 남긴다. 보정 DB에 "이 단계는 생략 가능/필수"류 항목만 쌓이면 정상이다.

항목 예시 (워크플로 단계에 대한 보정):

```markdown
## 용서하는 것 (생략 가능)

### [ship: 배포 절차]
- 규칙: 이 저장소는 main 직접 커밋 — PR 생성·VERSION 파일 단계 생략 가능 (버전은 plugin.json)
- 근거: 2026-07-06 /ship 표준 절차가 저장소 관행과 불일치, 축소 실행으로 무사고
- 신뢰도: 낮음(1)

## 벌주는 것 (반드시 한다)

### [ship: 테스트 게이트]
- 규칙: 소켓 계층 변경 시 통합 테스트는 생략하지 않는다
- 근거: 2026-07-06 핸드셰이크 변경 검증에 통합 테스트가 유효했음
- 신뢰도: 낮음(1)
```

## 응용 — 게이트형 자동화 파이프라인과 상호보완 (예: bkit)

[bkit](https://github.com/popup-studio-ai/bkit-claude-code) 같은 게이트형 자동화 파이프라인은 절차형 스킬 모음의 강화판이다: 단계를 나열하는 데서 그치지 않고 PDCA 상태 머신 · 고정 임계값 품질 게이트(matchRate ≥ 90% 등 11종) · 자동 수리 루프 · feature당 문서 산출(PRD/plan/design/analysis/report)로 절차를 **강제**한다. 자동화 수준을 정하는 설정(`/control` L0~L4)도 있지만 이 설정이 조절하는 축은 nunchi와 다르다 — 그래서 겹치지 않고 보완한다.

| | 게이트형 파이프라인 (bkit 등) | nunchi |
|---|---|---|
| 조절하는 축 | 자율성 — 어느 단계까지 무인 진행하고 언제 사용자에게 물을지 | 강도 — 각 단계가 이 환경에서 가치 있는지, 얼마나 깊게 할지 |
| 게이트 | 고정 임계값 — 전 환경 공통 | 환경별 예외 학습 — 게이트가 반복적으로 유효/무의미했던 기록 |
| 축적 지표 | Trust Score — 자동화 전반에 대한 신뢰 누적 점수 | 보정 DB — 단계별 판단 근거, 반증 가능한 규칙 |
| memory의 용도 | 진행 상태 영속화 — 세션이 끊겨도 재개 | 판단 근거 영속화 — 예측 어긋남만 |
| 실패 모드 | 모든 프로젝트에 전체 파이프라인 → 과잉 절차·문서 | 근거 없는 생략 → 반전 규칙으로 방지 |

**응용 방법:**

1. **자율성과 강도는 직교 — 함께 조여진다.** `/control level 4`(완전 자동)로 올려도 실행되는 절차의 **양**은 그대로다: 9개 phase, 게이트 11종, 문서 5종이 전부 돈다. 자율성 설정은 "누가 승인하는가"만 정하고 "무엇이 할 가치가 있는가"는 정하지 않는다. 보정 DB가 그 두 번째 축을 공급한다 — 이 환경에서 반복적으로 무의미했던 phase는 "용서하는 것"을 근거로 축소를 제안하고, 유효했던 게이트는 "벌주는 것"으로 자동화 수준과 무관하게 지킨다.
2. **고정 임계값의 환경별 예외로.** 게이트 임계값은 전 프로젝트 공통이라 정적 사이트에 dataFlow 7-hop 검사 같은 부적합이 생겨도 파이프라인 스스로는 학습하지 못한다. 특정 게이트가 이 환경에서 잡은 것이 없었는지 / 실제 회귀를 잡았는지가 정확히 보정 DB에 기록할 만한 내용이다.
3. **Trust Score와 중복 기록하지 않는다.** Trust Score는 "자동화를 얼마나 믿을 수 있나"의 단일 누적 점수이고, 보정 DB는 "어느 단계가 왜 과잉/필수였나"의 단계별 근거다. 진행 상태·작업 이력은 파이프라인의 memory/Task Management에, **강도 판단만** 보정 DB에 남긴다.

항목 예시 (파이프라인 단계에 대한 보정):

```markdown
## 용서하는 것 (생략 가능)

### [pdca: 문서 산출물]
- 규칙: 내부 도구 feature는 PRD+design 요약 1장으로 축소 가능 (5종 전체 산출 불필요)
- 근거: 2026-07-06 3파일 feature에 문서 작성이 구현보다 오래 걸렸고 이후 아무도 읽지 않았음
- 신뢰도: 낮음(1)

## 벌주는 것 (반드시 한다)

### [iterate: match 게이트]
- 규칙: 스키마 변경 feature는 gap-detector 검증을 생략하지 않는다
- 근거: 2026-07-06 matchRate 게이트가 설계-코드 불일치(누락 필드)를 실제로 잡았음
- 신뢰도: 낮음(1)
```

## 응용 — 방법론 스킬 모음과 상호보완 (예: superpowers)

[superpowers](https://github.com/obra/superpowers)는 "일하는 방식" 자체를 스킬로 만든 모음이다. 기능을 만들기 전에 brainstorming, 구현 전에 테스트 먼저(TDD), 버그가 나면 원인 추적부터(systematic debugging), 완료 선언 전에 검증 — 이런 작업 규칙 십여 개가 각각 스킬 문서로 들어 있다. 특징은 강제성이다: "적용될 가능성이 1%라도 있으면 반드시 해당 스킬을 발동하라"는 메타 규칙이 있고, 각 스킬에도 "실패하는 테스트 없이 프로덕션 코드를 쓰지 말라"(TDD의 Iron Law) 같은 예외 없는 조항이 있으며, "이번 한 번만 건너뛰자"는 생각 자체를 합리화로 규정해 차단한다.

이 설계에는 빈 곳이 두 개 있다:

1. **모든 프로젝트에 같은 강도로 적용된다.** 3줄짜리 일회성 스크립트에도 red/green 전체 사이클이 요구된다. 예외를 두는 방법은 "사용자에게 물어보라"뿐이고, 물어본 답은 어디에도 저장되지 않아 다음 세션에 같은 질문을 반복한다.
2. **경험이 다음 세션으로 이어지지 않는다.** 이 프로젝트에서 어떤 규칙이 실제로 사고를 막았고 어떤 규칙이 과했는지를 기억하는 장치가 없다. 스킬 내용은 저자가 새 버전을 릴리스할 때만 바뀐다.

nunchi가 채우는 것이 정확히 이 두 곳이다: 규칙이 이 환경에서 실제로 값을 했는지를 보정 DB에 기록하고, 그 기록을 세션 시작과 매 메시지마다 자동으로 불러온다.

| | superpowers | nunchi |
|---|---|---|
| 정하는 것 | 무엇을 지켜야 하는가 (작업 규칙) | 그 규칙을 이 프로젝트에서 얼마나 깊게 적용할 것인가 |
| 갱신 시점 | 저자가 새 버전을 릴리스할 때만 | 이 프로젝트에서 예측 어긋남을 겪을 때마다 |
| 예외 처리 | 매번 사용자에게 질문, 답은 휘발 | 판단을 기록해 다음 세션부터 자동 반영 |
| 세션 간 기억 | 없음 — 매 세션 같은 규칙을 다시 주입 | 보정 DB(코어 상시 주입 + 관련 항목 자동 회수) |

**응용 방법:**

1. **스킬 발동은 superpowers 규칙대로, 깊이는 nunchi로 조절한다.** 예: TDD 스킬은 발동하되, 이 프로젝트에서 일회성 스크립트에 전체 사이클이 반복적으로 과했다는 "용서하는 것" 기록이 있으면 실행 확인 1회로 축소한다. 단 "brainstorming은 항상 생략"처럼 발동 자체를 끄는 항목은 만들지 않는다 — 그것은 강도 조절이 아니라 규칙 폐기다.
2. **충돌하면 규칙을 따르고 결과를 기록한다.** "용서하는 것" 항목은 신뢰도 높음(3+)이 되기 전까지 가설이다. superpowers의 명시 조항과 부딪히면 일단 조항대로 실행하고, 그것이 과잉이었는지를 관찰해 항목을 갱신한다. 반대로 생략을 따랐다가 사고가 나면 반전 규칙에 따라 즉시 "벌주는 것"으로 옮긴다.
3. **기록할 곳을 구분한다.** 스킬 문서 자체의 개선은 superpowers의 몫이다 (writing-skills라는 자체 개선 스킬이 있다). 보정 DB에는 "이 프로젝트에서 이 규칙의 어느 부분이 과잉/필수였다"는 판단만 남긴다.

항목 예시 (방법론 규범에 대한 보정):

```markdown
## 용서하는 것 (생략 가능)

### [tdd: 일회성 스크립트]
- 규칙: scripts/ 하위 일회성 스크립트는 red/green 전체 사이클 대신 실행 확인 1회로 대체 가능
- 근거: 2026-07-06 3줄 스크립트에 TDD 사이클이 본체 작업보다 오래 걸렸음
- 신뢰도: 낮음(1)

## 벌주는 것 (반드시 한다)

### [debugging: 훅 오작동]
- 규칙: hook 관련 버그는 systematic-debugging의 근본 원인 추적을 생략하지 않는다
- 근거: 2026-07-06 증상 패치로 2회 재발, 근본 원인 추적으로 종결
- 신뢰도: 낮음(1)
```

## 응용 — 고정 강도 정책과 상호보완·충돌 (예: ponytail)

[ponytail](https://github.com/DietrichGebert/ponytail) 같은 고정 강도 정책 스킬은 워크플로 스킬 모음의 반대편 극단이다: 모든 환경에 "최소 강도"(YAGNI, stdlib 우선, 최단 diff)라는 고정 성향을 주입한다. gstack이 절차의 **상한**, ponytail이 강도의 **하한**이라면, nunchi는 그 사이 어디가 이 환경의 정답인지를 학습하는 층위다.

| | 고정 강도 정책 (ponytail 등) | nunchi |
|---|---|---|
| 접근 | 고정 성향 — 모든 환경에서 항상 최소 | 환경별 학습 — 예측 어긋남으로 갱신 |
| 방향 | 한 방향 (줄여라) | 양방향 (과잉·과소 모두 교정) |
| 자기교정 | 없음 — 틀린 환경에서도 유지 | 반전 규칙·신뢰도 승격 |
| 실패 모드 | 생략을 벌주는 환경에서 사고 반복 | 근거 없는 생략 → 반전 규칙으로 방지 |

**상호보완:**

1. **초기값으로.** 보정 DB가 비어 있는 신규 프로젝트에서 nunchi는 판단 근거가 없다. 고정 정책이 그 공백의 기본 성향이 되고, nunchi는 그 성향이 이 환경에서 어디서 틀리는지부터 기록하면 된다.
2. **맹점 보정으로.** 고정 정책의 최대 약점은 "이 환경에서 생략이 벌 받는 지점"을 모른다는 것이다. 그 지점이 정확히 "벌주는 것" 섹션이다. 정책을 따른 생략이 사고 나면 반전 규칙으로 즉시 기록되고, 다음 세션부터 같은 생략이 근거를 갖고 차단된다.
3. **재확인 기회로.** 정책과 "용서하는 것" 항목이 같은 생략을 가리키고 무사고로 끝나면, 해당 항목의 신뢰도 +1 관측 기회다. 정책이 보정 DB의 반증·재확인 시도를 늘려 학습을 가속한다.

**충돌과 우선순위:**

둘 다 활성이면 매 판단마다 정책이 당기는 방향(항상 최소)과 보정 DB가 가리키는 방향이 어긋날 수 있다. 해소 규칙:

- **"벌주는 것" 항목은 고정 정책보다 우선한다.** 특히 신뢰도 높음(3+)은 이 환경의 확정 규칙이다. 정책이 "테스트도 YAGNI"라 해도, 이 환경이 테스트 생략을 벌준 기록이 있으면 한다.
- 신뢰도 낮음(1) 항목과 정책이 충돌하면 항목을 가설로 취급하되, 이번 결과를 근거로 항목을 갱신한다 — 어느 쪽이 맞았는지가 곧 관측이다.
- 어느 쪽도 근거가 없으면(빈 문서 + 정책) 정책을 따르고, 결과가 예측과 어긋나면 기록한다. 충돌 자체가 보정 기록 축적의 입력이다.
- 사용자의 명시적 지시는 언제나 둘 모두에 우선한다 (SKILL.md의 기존 규칙과 동일).

**우선순위 핸드셰이크 (자동):**

위 해소 규칙 중 "어느 쪽이 기본인가"는 nunchi가 정하지 않고 **사용자에게 위임**한다. SessionStart hook이 `enabledPlugins`에서 ponytail 활성을 감지하면 `policy-priority` 상태에 따라 컨텍스트에 한 줄을 주입한다:

1. **미결정(기본)**: "작업 강도 판단이 처음 충돌하면 사용자에게 물어보라"는 지시를 주입. 실제 충돌 시점에 AskUserQuestion으로 우선순위를 묻고, 답을 프로젝트 `.claude/nunchi.json`의 `policy-priority`에 저장한다. 충돌이 없는 세션에서는 묻지 않는다.
2. **`nunchi`**: 충돌 시 보정 DB 우선 규칙 1줄만 주입.
3. **`ponytail`**: 충돌 시 최소화 우선 규칙 1줄만 주입. 단 신뢰도 높음(3+)의 "벌주는 것" 항목은 실제 사고 기록이므로 예외로 지킨다.

저장된 결정은 다음 세션부터 hook이 읽어 자동 반영된다 (재로드 불필요). ponytail이 비활성이면 아무것도 주입하지 않는다 — 비용 0. 결정을 바꾸려면 nunchi.json의 값을 수정하거나 키를 지워 재질문 상태로 되돌린다.

항목 예시 (고정 정책에 대한 보정):

```markdown
## 벌주는 것 (반드시 한다)

### [최소화: 소켓 계층]
- 규칙: 소켓/동시성 코드는 최단 diff보다 경쟁 케이스 방어를 우선한다
- 근거: 2026-07-06 최소 구현이 동시 스폰 포트 경쟁에서 실패, 락 확인 코드가 필요했음
- 신뢰도: 낮음(1)

## 용서하는 것 (생략 가능)

### [최소화: 내부 헬퍼]
- 규칙: hooks 내부 헬퍼는 정책대로 최소 구현 유지 — 방어 코드 불필요
- 근거: 2026-07-06 입력이 Claude Code hook payload로 고정되어 방어가 무의미했음
- 신뢰도: 중간(2)
```

**중복 기록은 피한다.** 정책의 일반 원칙("stdlib 우선" 등)은 보정 DB에 옮겨 적지 않는다 — 그건 정책 스킬이 이미 매 세션 주입한다. 보정 DB에는 **정책이 이 환경에서 틀리거나 맞았던 지점**만 남긴다.

## 튜닝

- 점검 주기: 환경변수 `NUNCHI_CHECK_EVERY` (기본 10, 최소 2)
- 정제 기준: SKILL.md 정제 규칙의 60건 기준을 직접 수정
- 자동 주입 건수: `hooks/user-prompt-submit.ts`의 `limit: 3`

/plugin uninstall nunchi@devp0tion
